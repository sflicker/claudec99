# stage-70-02 Transcript: Line and Column Tracking in Tokens

## Summary

Stage 70.02 introduces comprehensive source position tracking to the ClaudeC99 compiler by adding line and column fields to every token. A new `SourceFile` struct (containing the file path) tracks which file each token came from. The lexer maintains position state (current line, column, and file) and records the position at the start of each token. Location markers—generated by the preprocessor at include boundaries—update the lexer's file and position state when transitioning between files. The `--print-tokens` output format is extended with a new `[line:col]` column positioned after the token index, with width dynamically sized to zero-pad values to the minimum digits needed for the largest line and column in the input.

## Changes Made

### Step 1: Token Structure and File Tracking

- Added `SourceFile` struct to include/token.h with single field: `char *path`
- Added three fields to `Token` struct: `int line`, `int col`, `SourceFile *file`
- All token positions are 1-based (line 1, column 1 at the start of a file)

### Step 2: Lexer Position State and Initialization

- Updated include/lexer.h to add position fields to `Lexer`: `line`, `col`, `file`, and three fields for SourceFile pool management (`file_pool`, `n_files`, `files_cap`)
- Modified `lexer_init()` signature to accept `filename` parameter: `lexer_init(Lexer*, const char *source, const char *filename)`
- Added `lexer_free(Lexer*)` function to release the SourceFile pool
- Implemented `lexer_init()` to initialize line=1, col=1, and intern the file by path using a new `lexer_intern_file()` helper

### Step 3: Lexer Internals and Position Advancement

- Implemented `lexer_intern_file()` to manage a per-lexer pool of `SourceFile` objects, looked up by path string (with deduplication)
- Implemented `lexer_advance()` to advance the position counter while updating `line` and `col` state: increments `col` for regular characters, resets `col` to 1 and increments `line` when consuming a newline
- Implemented `token_set_pos()` helper that records the lexer's current `line`, `col`, and `file` into a token

### Step 4: Location Marker Parsing

- Implemented `lexer_parse_location_marker()` in src/lexer.c to parse preprocessor-emitted location markers with format `\x01<line>:<path>\n`
- Updated lexer position state (file and line/col) when encountering a marker
- Modified `lexer_skip_whitespace()` to detect and handle `\x01` markers (not skip them)
- Updated `lexer_next_token()` to call `lexer_parse_location_marker()` when whitespace skip encounters a SOH byte

### Step 5: Preprocessor Location Markers

- Modified src/preprocessor.c at the `#include` directive handling site to inject two location markers:
  1. Before included content: `\x01 1:<include_path>\n` (enter-file marker, resets to line 1)
  2. After included content: `\x01 <current_line>:<source_path>\n` (return-to-parent marker, uses preprocessor's tracked line number)
- Return-to-parent marker line number accounts for the `\n` from the `#include` directive itself (which naturally increments the lexer's line counter by 1)

### Step 6: Token Position Recording

- Updated `lexer_next_token()` in src/lexer.c to call `token_set_pos()` immediately after whitespace skip and location marker parsing, capturing the lexer's position before consuming token characters
- Replaced all bare `lexer->pos++` and `lexer->pos += N` increments with `lexer_advance()` calls throughout the tokenizer

### Step 7: Print-Tokens Output Formatting

- Updated `print_token_row()` in src/compiler.c to accept `line_width` and `col_width` parameters
- Format new position column as `[%0*d:%0*d]` using the provided widths, placed immediately after the token index column
- Updated `print_tokens_mode()` to:
  - Accept `source_file` parameter and pass it to `lexer_init()`
  - Compute `max_line` and `max_col` across all tokens in the stream
  - Derive formatting widths via `digit_width()` function
  - Call `lexer_free()` to clean up the SourceFile pool
  - Pass widths to all `print_token_row()` calls

### Step 8: Parser and Compiler Integration

- Updated `lexer_init()` call in the parse path (src/compiler.c) to pass the `source_file` parameter
- Updated call to `print_tokens_mode()` to pass the `source_file` parameter

### Step 9: Test Regeneration

- Regenerated all 99 print-tokens expected files (test/print_tokens/*.expected) to include the new `[line:col]` column with appropriate zero-padding

## Notes

- Location marker format uses SOH control byte (`\x01`) as a sentinel because it cannot appear in valid C source, avoiding any conflict with C syntax
- Return-to-parent marker line number uses the preprocessor's tracked line number (not +1) because the `\n` from the `#include` directive line remains in the preprocessed output and naturally increments the lexer's line counter
- SourceFile pool is owned by the lexer and looked up by path string for deduplication; short-lived for the parser path (not referenced by AST), so no explicit free needed after parsing except for print-tokens mode
- Position column width is computed once per token stream based on the maximum line and column values observed, ensuring consistent zero-padding across all tokens
